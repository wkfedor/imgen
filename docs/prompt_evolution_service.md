# Сервис эволюции промптов

## Основная идея

Сервис нужен для эволюционной разработки промптов вместе с пользователем и ИИ. Нормальный цикл: одна версия промпта, одна выбранная модель, один результат или небольшой batch именно для этой модели, затем обратная связь и новая дочерняя версия промпта.

Это должно ощущаться как Git-история для работы с промптами:

- каждая версия промпта сохраняется;
- каждая сгенерированная картинка привязана к точной версии промпта и модели;
- каждая критика сохраняется;
- можно делать ветки;
- пользователь и ИИ могут продолжить работу из середины дерева;
- работа не должна зависеть только от памяти диалога.

## Обязательный workflow

### Один промпт, одна модель

Каждый прогон должен использовать одну выбранную модель, если пользователь явно не попросил сравнить модели. Если тот же промпт нужно проверить на другой модели, это отдельный прогон от той же версии промпта.

Пример:

```text
revision v3 + Lunerria
revision v3 + RealVisXL
выбираем направление результата
продолжаем от выбранного узла как v4
```

### Дерево промптов

Эволюция промпта - это дерево, а не только линейный список.

```text
v1 исходный промпт
  v2 больше мультяшности
    v3 сильнее координаты полок
  v2b та же идея на RealVisXL
    v3b лучше структура, хуже стиль
```

Пользователь и ИИ могут выбрать любой узел и продолжить от него.

### Сначала обратная связь

Перед созданием следующего промпта нужно записать обратную связь по выбранной картинке:

- что получилось хорошо;
- что получилось плохо;
- что сохранить;
- что убрать;
- куда двигаться дальше;
- заметки по поведению конкретной модели;
- мнение ИИ;
- мнение пользователя.

Следующий промпт должен выводиться из этой обратной связи, а не сочиняться заново.

## Дисциплина summary

Каждый значимый шаг нужно сохранять в summary-файл, потому что длинные генерации могут зависать, падать или терять контекст диалога.

Активный summary-файл:

```text
tmp/prompt_evolution_summary.md
```

Summary должен включать:

- текущую цель;
- текущий узел дерева промптов;
- выбранную модель;
- параметры генерации;
- путь к сгенерированной картинке;
- статус результата;
- обратную связь пользователя;
- оценку ИИ;
- следующий запланированный шаг;
- блокер, если он есть.

## Черновая модель данных

### PromptProject

Одна творческая задача.

Поля:

- title;
- original_goal;
- acceptance_criteria;
- status;
- active_revision_id.

### PromptRevision

Одна версия промпта.

Поля:

- prompt_project_id;
- parent_revision_id;
- version_label;
- prompt;
- negative_prompt;
- change_summary;
- created_from_feedback_id.

### PromptRun

Один прогон одной версии промпта на одной модели.

Поля:

- prompt_revision_id;
- model_name;
- width;
- height;
- steps;
- seed;
- status;
- image_result_id;
- image_path;
- started_at;
- finished_at.

### PromptFeedback

Критика и направление для одного прогона.

Поля:

- prompt_run_id;
- positives;
- negatives;
- keep;
- remove;
- next_direction;
- ai_evaluation;
- user_evaluation;
- selected_for_continuation.

## MVP-поведение

1. Создать проект и исходный промпт.
2. Запустить одну версию промпта на одной выбранной модели.
3. Показать картинку и спросить feedback пользователя.
4. ИИ записывает свою оценку.
5. Пользователь и ИИ решают, стоит ли продолжать от этого узла.
6. Создаётся новая дочерняя версия промпта.
7. Дочерняя версия запускается на выбранной модели.
8. Цикл повторяется.

Сравнение моделей допустимо, но только как отдельные прогоны от одного и того же узла промпта.

## Урок из эксперимента со стеллажом

Ранний workflow съехал в массовый прогон нескольких моделей и скрытые приватные генерации. Это не то поведение, которое нужно продукту.

Правильное поведение:

- использовать видимую историю сервиса;
- сохранять каждую пару промпт/картинка;
- если тестируются несколько моделей, записывать их как отдельные ветки/прогоны;
- после результатов останавливаться и спрашивать пользователя, какую ветку продолжать;
- всегда обновлять summary до и после генерации.
